My Upgraded API Pentester Plan

This is your actionable career plan to become a professional-grade API Penetration Tester. It is a $0 Budget Plan focused on project-based deliverables that prove your skill. This plan is built on the "Build, Break, Secure" philosophy to make you a non-replaceable, high-value candidate.

Why am I doing this?

To fix your self-admitted weakness. This phase builds the "terminal muscle memory" required to operate in the headless (non-GUI) Linux environments where all professional targets (servers, APIs, containers) reside.

What:

Mastering core Linux and Git commands until they are muscle memory.

How to Practice ($0):

  • Linux (Practice): Finish all levels of OverTheWire: Bandit. This is your #1 priority for core commands.
  • Git (Practice): Create a GitHub repo. Practice the full workflow: git clone, git add ., git commit -m "msg", git push. Do this 20 times.
  • Git (Learn): Use the Git Tower (Free Book) for a clear guide if you get stuck.

Deliverable:

A GitHub repository with a 100-commit history (even if just text files) to build muscle memory.

✅ Definition of Done: You can finish a Bandit level and `git push` your notes without looking up the commands.

What:

Turning a simple Bash one-liner into a professional, reusable Python tool. This demonstrates "Text-Fu" and practical Python scripting.

How to Practice ($0):

  • Service Mgmt (Practice): On your Debian VM, install Nginx: sudo apt install nginx. Control it: sudo systemctl status nginx.
  • Log Analysis (Learn): Read this DigitalOcean `grep`/`awk`/`sed` Guide. This is your textbook for the concept.
  • Log Analysis (Lab): Use the [Free Room] TryHackMe 'Tinkerbell' Room. It is a specific free lab for practicing log analysis.
  • Python (Learn): Learn the `argparse` module for CLI arguments and the `re` module for regex parsing.

Deliverable:

A new GitHub repo log-analyzer.py. This tool must:
1. Use argparse to accept a file path (-f).
2. Use Python's re (regex) to parse the Nginx log.
3. Output structured JSON (e.g., top 5 failed IPs).

✅ Definition of Done: Your script runs from the command line, parses a real /var/log/nginx/access.log, and prints a clean JSON report.

Why am I doing this?

To build your core specialization. This phase pivots from generic web hacking to the high-demand field of API-specific vulnerabilities.

What:

Master the OWASP API Top 10. Your primary focus must be on Business Logic Flaws: API1 (BOLA), API5 (BFLA), and API2 (Broken Authentication).

How to Practice ($0):

  • Study (Learn): Read the OWASP API Top 10 Project documentation for each vulnerability.
  • Practice (Mandatory): Complete all labs in the PortSwigger Academy "API Security" module. This is your new bible.
  • Practice (Tools): Install Postman and Burp Suite Community. Use them to manually poke at the PortSwigger labs.
  • Practice (Lab): Explore the vAPI (Vulnerable API) project as an extra target.

Deliverable:

A document in your GitHub repo (API_NOTES.md) explaining BOLA, BFLA, and Broken Authentication in your own words.

✅ Definition of Done: You have completed all Practitioner-level API security labs in the PortSwigger Academy and can explain BOLA by heart.

What:

Building an extensible Python toolkit, not just a one-off script. This shows you build frameworks, not just hacks.

How to Practice ($0):

  • Learn (Requests): Read the official requests library docs.
  • Project: Build a reusable Python script (api-toolkit.py) that uses argparse to select a "module".
    - Start with module_bola (iterates IDs).
    - Add module_discovery (takes a wordlist, finds hidden endpoints).

Deliverable:

A new GitHub repo api-pentest-toolkit containing your finished, modular tool. The README.md must explain how to use it and how to add new modules.

✅ Definition of Done: You have a working, reusable Python script that can run at least two different "attack modules" (e.g., BOLA and Discovery) selected via CLI arguments.

Why am I doing this?

To validate your skills on real infrastructure without competing against elite hackers. As the strategy suggests, we will ignore paid public programs for now and focus on VDPs to build a portfolio.

What:

Finding targets that aren't saturated by automation.

How to Practice ($0):

  • Avoid the Front Page: Do not pick the top programs on HackerOne. They are "cleaned by hackers doing this for 5 years".
  • Find VDPs: Search for companies with "Vulnerability Disclosure Programs" that offer "Thanks" or "Swag" but not cash.
  • Self-Hosted: Look for "self-managed" programs (on the company's own website) rather than those hosted on big platforms.

Deliverable:

A list of 3 low-competition VDP targets that look interesting to you.

✅ Definition of Done: You have selected ONE specific VDP target for the next step.

What:

Applying "Pattern Recognition" to a single target over a sustained period.

The Routine:

  • The Rule: Stick to ONE target for 2 weeks. Do not jump around.
  • Daily Schedule: 2 hours of studying a feature you don't understand, followed by 2-3 hours of active hunting on that feature.
  • Focus: Map the application's business logic. Look for where patterns (like IDOR/BOLA) should exist based on your PortSwigger training.

Deliverable:

A valid vulnerability report (even a P4/Low) or a detailed "Attack Surface Map" of the target if no bugs are found.

✅ Definition of Done: You have spent 14 days on one target and understand its business logic better than the developers.

Why am I doing this?

To learn the "Build It" part of the "Build, Break, Secure" philosophy. You will build your own vulnerable target to understand a developer's logic, which is essential for finding logic flaws.

What:

Use Python Flask to create a REST API with /login and user endpoints. This will be your target. We will *deliberately* code multiple flaws.

How to Practice ($0):

  • Learn: Use the official Flask "Quickstart" docs and RealPython's Flask API guides.
  • Flaw 1 (BOLA): The /users/<id> endpoint must *not* check if the logged-in user matches the <id>.
  • Flaw 2 (BFLA): Create an /admin/all_users endpoint that *only* checks for login, not for an "admin" role.
  • Flaw 3 (Mass Assignment): Create a PATCH /profile endpoint that naively updates a user object, allowing a user to "promote" themselves by sending {"is_admin": true}.

Deliverable:

A new GitHub repo, multi-vuln-api-lab, containing the full source code for your API with all three vulnerabilities.

✅ Definition of Done: Your API runs locally and is demonstrably vulnerable to BOLA, BFLA, and Mass Assignment.

What:

Package your vulnerable Flask API using Docker to simulate a real-world production environment.

How to Practice ($0):

  • Learn: Use Docker's "Get Started" guide.
  • Practice (Lab): Use Play with Docker for a free, in-browser lab.
  • Project: Write a Dockerfile for your Flask app. Build it (docker build) and run it (docker run). Add a docker-compose.yml file for easy startup.

Deliverable:

A Dockerfile and docker-compose.yml file committed to your multi-vuln-api-lab repo.

✅ Definition of Done: Your vulnerable API runs successfully from a Docker container using a single docker compose up command.

Why am I doing this?

To learn the "Break It" part. You will now write a custom, logic-based exploit for the target you built, and *document* it like a professional.

What:

Adapt your api-pentest-toolkit (or write a new exploit.py) to find all three vulnerabilities in your custom lab.

How to Practice ($0):

  • Project: This is a pure project-based step. You will use your own code from Phase 1 and 2.
  • Script: Your tool must login as a low-privilege user_A.
  • Exploit: It must then use that user's token to:
    1. Exploit BOLA to read data for user_B.
    2. Exploit BFLA to access the /admin/all_users endpoint.
    3. Exploit Mass Assignment to make user_A an admin.

Deliverable:

A detailed README.md in the multi-vuln-api-lab repo, with an "Exploitation Steps" section and the finished exploit.py script.

✅ Definition of Done: Your exploit.py script, when run, successfully exploits all three vulnerabilities.

What:

Creating a high-quality PDF report for your findings. This is a non-technical skill that is critical for getting hired. You are proving you can do the *job*, not just the *hack*.

How to Practice ($0):

  • Template (Learn): Search "pentest report template github" to find free Markdown or DocX templates.
  • Writing (Practice): Write a professional report for your multi-vuln-api-lab project.
  • Content: It must include:
    1. Executive Summary: Simple, non-technical summary for a manager.
    2. Technical Findings: One section for each of your 3 bugs (BOLA, BFLA, Mass Assignment).
    3. Proof of Concept: Your exploit.py code/output.
    4. Remediation: Clear, actionable advice on how to *fix* the code.

Deliverable:

A high-quality Vulnerability_Report.pdf, exported and committed to the multi-vuln-api-lab repo. Link this PDF directly on your CV.

✅ Definition of Done: You have a professional PDF report that clearly explains the risk, impact, and solution for all three vulnerabilities.

Why am I doing this?

To tackle API2: Broken Authentication. JSON Web Tokens (JWT) are the #1 auth method for modern APIs and a massive attack surface. This becomes a *second* major portfolio project.

What:

Use Flask and the PyJWT library to build an API, deliberately introducing common JWT flaws.

How to Practice ($0):

  • Build (Part 1): Create a new Flask API with /login (issues a JWT) and /protected (requires a JWT).
  • Flaw 1 (alg:none): Deliberately code the /protected endpoint to *accept* tokens with the alg header set to none.
  • Flaw 2 (Weak Secret): Use a laughably weak secret key, like secret123.
  • Break (Part 2): Write a Python script (jwt_attacker.py) using requests and pyjwt to:
    1. Forge a token with alg: none and access /protected.
    2. Brute-force the weak secret key using a simple wordlist.

Deliverable:

A new GitHub repo jwt-security-lab with the vulnerable Flask API source and the jwt_attacker.py exploit script.

✅ Definition of Done: Your jwt_attacker.py script can successfully get admin access by exploiting both the alg:none and weak secret vulnerabilities.

What:

Fixing the flaws you introduced, demonstrating the "Secure" part of the cycle.

How to Practice ($0):

  • Secure (Part 1): Create a new branch patched.
  • Fix 1: Update the code to *only* accept the HS256 algorithm, rejecting all others.
  • Fix 2: Change the key to a long, complex, random string.
  • Verify: Run your jwt_attacker.py script against the patched code. It must fail.

Deliverable:

The patched branch in your repo, with a README.md explaining the vulnerabilities and the fixes.

✅ Definition of Done: Your exploit script works on the main branch but fails completely on the patched branch.

Why am I doing this?

To learn the "Secure It" part and tie everything into one "Resume Weapon." This single project proves you are a professional who understands the full "Build, Break, Secure" lifecycle.

What:

Use GitHub Actions to create a CI/CD pipeline that automatically runs your exploit script as a security test against your multi-vuln-api-lab.

How to Practice ($0):

  • Learn: Read the GitHub Actions Docs.
  • Tool (Optional): Use act (GitHub Tool) to run your Actions locally for faster testing.
  • Step 1 (Build Action): Create .github/workflows/security_test.yml. It must build your Docker container, wait 10s, and then run python exploit.py.
  • Step 2 (The "Break"): Push your vulnerable code. The exploit.py will succeed, and your pipeline must fail (turn RED).
  • Step 3 (The "Secure"): Create a new branch patched. Fix all 3 flaws (BOLA, BFLA, Mass Assignment) in your Flask code.
  • Step 4 (The "Proof"): Push your patched code. The exploit.py will fail to find the bugs, and your pipeline must pass (turn GREEN).

Deliverable:

The "Resume Weapon": The single GitHub URL for your multi-vuln-api-lab repo, showing a clear history of RED (vulnerable) and GREEN (patched) builds.

✅ Definition of Done: You have a GitHub Action that turns RED on your main branch and GREEN on your patched branch. This is the ultimate proof of your skills.
